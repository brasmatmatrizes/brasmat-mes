# BRASMAT MES

Sistema de execução de manufatura (MES) para a Brasmat — ferramentaria de matrizes de metal duro. Acompanha o caminho de cada peça pela produção (retífica, torno, erosão, etc.), do lançamento à expedição.

## Arquitetura

- **Front-end estático puro** — sem build, sem servidor próprio. Cada página é um `.html` autocontido (HTML + CSS inline em `<style>` + JS inline em `<script>`).
- **`status.js`** — lógica central compartilhada por todas as páginas: fetch ao Supabase, `status()`, helpers de progresso/tipo, e o painel de linha do tempo (`abrirPainelDetalhe`).
- **Supabase** (PostgreSQL na nuvem) é o backend. O front usa **só a chave pública (publishable)** definida no topo do `status.js` (`SUPA_URL`, `SUPA_KEY`). Project ref: `hjvlznijsgdwurtsyukl`.
- Deploy automático via **Vercel** (`brasmat-mes.vercel.app`) a cada push no `main` — **mas o webhook já falhou silenciosamente uma vez (18/07/2026)**: push confirmado no GitHub, Vercel não buildou nada, site ficou no ar com a versão antiga sem nenhum erro visível. **Sempre confirmar o deploy** (ver Convenções de trabalho), nunca assumir que push = no ar. Após mudança confirmada, lembrar de Ctrl+F5 (cache).
- **Consumo de dados (egress)** é a única cota do plano gratuito que aperta. Antes de mexer em qualquer coisa que altere volume de fetch/Realtime, ler `docs/otimizacao-egress-jul2026.md` — traz as medições, o padrão obrigatório do canal Realtime e os caminhos já descartados (com o motivo). Para medir/diagnosticar, usar a skill `verificar-uso-supabase`.

Um item cadastrado pode gerar **1+ lotes por posição** (`material_lote_componentes`, cada linha já é um lote de uma posição só — `nucleo`/`carcaca`/`luva`, núcleo sempre, carcaça/luva opcionais — numerado automaticamente **por posição**, não por item: o núcleo pode ter 2 lotes enquanto a carcaça tem só 1, independentes, revisão 19/07/2026). Cada lote/componente tem o **material real daquela posição** (`metal_duro` ou `aco` — não é fixo; núcleo pode ser dos dois, carcaça/luva são sempre aço) e sua própria quantidade. **Um cartão do kanban = um lote**, pra não travar dois processos diferentes num cartão só. A criação vem sempre da **Mesa de Planejamento** (dentro da própria tela, aberta pela busca) — uma seção independente por posição ativa, cada uma com sua própria barra de alocação, mini-trilha clicável (fato = já aconteceu / intenção = "entra por", só enquanto Aguardando) e divisão/duplicação de lotes. Arrastar o cartão pra uma coluna **registra um evento** (`material_eventos`: um registro por movimento, com a data — nunca editado, nunca texto livre) — status sempre calculado do último evento (view `material_ultimo_evento`), nunca digitado. Clicar (sem arrastar) abre histórico + "Outros lotes deste item" (todas as posições) + botão excluir. Card e painel mostram o resumo do item inteiro (ex. "⚠ 1 de 2 lotes prontos") via `materialResumoItem` (`status.js`, compartilhada com o Cadastro Roteiro) — completude é **por item** (todos os lotes de todas as posições prontos), não depende de pareamento núcleo↔carcaça↔luva. Sem fornecedor/preço/saldo de estoque nesta fase (fora do escopo, ver spec).

Ao criar o roteiro a partir de um item, copia OS/qtd/tipo/cliente/prazo do item. Um roteiro por **pedido+código** (`roteiros`); paleta de operações do catálogo (`roteiro_operacoes`) agrupada por família reordenável (`roteiro_familias`) — um clique adiciona na sequência (`roteiro_itens`); subcomponente por linha (lista em `roteiro_subcomponentes`, chips definem o das novas linhas); tudo com salvamento imediato/debounce. ⚙ abre manutenção do catálogo e subcomponentes. 🖨 gera impresso (layout tipo Excel, agrupa por subcomponente) com QR Code do **link do Forms pré-preenchido montado automaticamente** (`montarLinkForms`: base + `FORM_ENTRY.*` do Forms "PILOTO 2 REV.A"; preenche razão social, classif. fiscal, código, pedido, OS, qtd solicitada, **qtd de operações = nº de linhas do roteiro**, cliente e prazo — só existe neste ponto porque só aqui o nº de operações é conhecido). ## Dados Supabase

- **`producao_eventos`** (tabela) — um registro por evento (Entrada/Saída) de cada peça em cada processo. Colunas-chave: `pedido`, `codigo_peca`, `processo`, `evento`, `data_evento`, `os`, `operador`, `cliente`, `prazo_entrega`, `quantidade_solicitada`, `quantidade_produzida`, `retrabalho`, `sequencial_operacao`, `quantidade_operacoes`, `razao_social`, `classificacao_fiscal`, `operacao_*` (subop por setor).
- **`posicao_atual`** (view) — posição atual (último evento) de cada peça, um registro por peça+pedido. **NÃO tem** `razao_social`, `classificacao_fiscal`, `sequencial_operacao` nem `quantidade_operacoes` — esses só existem em `producao_eventos`. Tem `suboperacao` (coluna calculada, criada 22/07/2026): resolve as 13 colunas `operacao_*` numa só (`coalesce` do primeiro valor não-vazio — só uma tem valor por linha). `buscarPosicao()` pede colunas explícitas (`POS_COLS` em status.js) e **deixa as 13 de fora** — corta o payload pela metade (782→405 kB/1000 linhas). As 13 continuam na view por compatibilidade; `getSubop()` usa `suboperacao` quando existe e mantém o percurso das 13 para linhas de `producao_eventos`, **que não têm essa coluna** — não remover esse fallback.
- **`historico_peca`** (view, criada 22/07/2026) — histórico de uma peça para a linha do tempo (`buscarHistorico`). Faz duas coisas **no servidor**: (1) `DISTINCT ON (pedido, codigo_peca, processo, evento, operador, data_evento)` descarta os **11.934 apontamentos duplicados** (script do Forms gravava 2x antes da correção do LockService; zero novos desde 04/07/2026 — é resíduo histórico, não bug ativo), que faziam a linha do tempo exibir o mesmo passo repetido em ~2 de cada 3 peças; (2) entrega só as 18 colunas usadas (de 33), com `suboperacao` já resolvida. Medido: 128→39 kB numa peça de 138 eventos. **Filtrar no front não serviria** — economiza banda só quem não envia. `producao_eventos` continua intacta.
- **`demanda_itens`** (tabela) — itens da Demanda Cliente. Colunas: `id`, `cliente`, `codigo_peca`, `pedido`, `os`, `descricao`, `data_desejada`, `prazo_confirmado`, `desenho_url`, `fluxo_url`, `concluido` (bool), `observacao`, `ordem` (int, ordenação dos cards), `exp_status` (almoxarifado: `pendente`/`separado`/`expedido`), `created_at`, `updated_at`. RLS liberado para `anon`.
- **`demanda_clientes`** (tabela) — personalização por cliente: `cliente` (PK = nome real), `nome_fantasia`, `cor` (hex). Usada na Demanda e na Expedição (cor de fundo + nome curto).
- **`kanban_operadores`** (tabela) — operadores selecionados para o Painel: `operador` (PK), `ordem`.
- **`operadores_distintos`** (view) — `select distinct operador from producao_eventos` (lista completa p/ a config do Painel).
- **`roteiro_operacoes`** (tabela) — catálogo de operações da Engenharia: `codigo` (único, ex. "2.2"), `descricao`, `familia`, `ativo` (bool). Carga inicial: 60 operações da planilha Excel.
- **`roteiros`** (tabela) — cabeçalho do roteiro, **unique(pedido, codigo_peca)**: `pedido`, `codigo_peca`, `os`, `quantidade`, `observacao` (orientações do impresso). Índices em `pedido` e `codigo_peca`.
- **`roteiro_itens`** (tabela) — linhas da sequência: `roteiro_id` (FK, on delete cascade), `sequencia`, `subcomponente`, `operacao_codigo`, `operacao_descricao` (snapshot — não muda se o catálogo mudar), `observacao`.
- **`roteiro_subcomponentes`** (tabela) — lista de subcomponentes possíveis: `nome` (único), `ordem`. Carga inicial: Inferior, Superior, Luva.
- **`roteiro_familias`** (tabela) — ordem dos grupos da paleta da Engenharia (o processista reordena com ◂▸): `nome` (único), `ordem`. Família nova entra automaticamente no fim (`ensureFamilia`).
- **`roteiro_clientes`** (tabela) — espelho do dropdown "Cliente" do Forms (usado no cadastro de peça nova): `nome` (único, ex. "5_CISER FIXADORES AUTOMOTIVOS S/A - SARZEDO", valor exato do Forms), `ordem`. Carga inicial: 28 clientes extraídos do HTML público do Forms. **Re-sincronizado sob demanda** (não mantido pela tela) — ver regra do QR do Forms.
- **`roteiros`** ganhou (06/07): `razao_social`, `classificacao_fiscal`, `cliente`, `prazo_entrega` (date), `forms_link` — cabeçalho do Forms guardado pra montar o QR pré-preenchido.
- **`itens`** (tabela, criada 17/07) — cadastro do Comercial (Cadastro Item), fonte da busca do Cadastro Roteiro: `pedido`, `codigo_peca`, `os`, `quantidade`, `razao_social`, `classificacao_fiscal` (derivados do Tipo V/I, mesma regra do `roteiros`), `cliente`, `prazo_entrega` (date), `observacao`, `ferramenta_interna` (bool, default false, criada 19/07 — isenta o item do bloqueio de material no Cadastro Roteiro), `plano_separacao_concluido` (bool, default false, criada 19/07 — PCP marca na Mesa de Planejamento que terminou de planejar a separação deste item). **unique(pedido, codigo_peca, os)** — identidade do item inclui a OS (permite mesmo código+pedido com OS diferente). Índice em `codigo_peca`.
- **`material_lote_componentes`** (tabela, criada 18/07, reestruturada 19/07 — `material_lotes` foi **apagada**, fundida aqui) — cada lote é uma posição física de um item: `item_id` (FK `itens.id`, direto), `posicao` (`nucleo`/`carcaca`/`luva` — núcleo sempre existe, carcaça/luva opcionais), `tipo_material` (`metal_duro`/`aco` — **não é fixo por posição**: núcleo pode ser dos dois, carcaça e luva são sempre `aco`), `lote_numero` (1,2,3... automático, **escopado por item+posição** — núcleo e carcaça numeram independente um do outro), `quantidade` (int, própria do lote), `observacao`, `entrada_prevista_ordem` (int, nullable — intenção de por onde o lote vai entrar na trilha, enquanto ainda não tem evento real). **unique(item_id, posicao, lote_numero)**. Cada lote = 1 cartão no kanban.
- **`material_estagios`** (tabela, criada 18/07, reestruturada 18/07) — catálogo único das colunas do kanban (trilha compartilhada, não mais por material): `descricao`, `ordem`, `aplica_metal_duro`/`aplica_aco` (bool — define quais cartões podem parar naquela coluna). **unique(ordem)**. Seed (5): Compra(MD+Aço) → Recebido/Estoque(Aço) → Usinagem(Aço) → Tratamento Térmico(Aço) → Pronto(MD+Aço). Cartão de Metal Duro pula direto de Compra pra Pronto.
- **`material_eventos`** (tabela, criada 18/07, reestruturada 18/07) — um registro por **movimento do cartão** (arrastar pra uma coluna): `componente_id` (FK `material_lote_componentes.id`), `estagio_id` (FK) + `estagio_descricao`/`estagio_ordem` (snapshot, não muda se o catálogo mudar), `data_evento`. Sem par Entrada/Saída (simplificado — um evento por drop já marca a transição); nunca editado, só inserido.
- **`material_ultimo_evento`** (view) — último evento (`distinct on componente_id`) de cada componente, mesmo padrão de `posicao_atual`. A coluna/status exibido é **calculado no front** a partir daqui, nunca digitado.
- **Bucket `demanda-arquivos`** (Storage, público) — anexos de desenho/roteiro (PDF **ou imagem** — `allowed_mime_types` liberado). Caminho: `CLIENTE/CODIGO/{desenho|fluxo}-{timestamp}.ext`. Ao remover anexo / trocar / remover item, o arquivo é apagado do bucket.
- **Realtime** habilitado (publication `supabase_realtime`) nas tabelas `producao_eventos` e `demanda_itens`. Kanban, Painel, Expedição, Apontamentos e Demanda usam `@supabase/supabase-js` (CDN) para atualizar instantaneamente. **Padrão obrigatório do canal** (revisão 22/07/2026, ver `AUDITORIA_TECNICA.md` P4): o callback do `subscribe` tem que tratar `CLOSED`/`CHANNEL_ERROR`/`TIMED_OUT` chamando `agendarReconexao()` (backoff 2s→30s) e, ao *re*conectar, refazer a busca uma vez para recuperar o que passou — sem isso a aba fica presa no `setInterval` de fallback (180s), gastando banda e mostrando dado atrasado **sem avisar ninguém**. O fallback é último recurso, não o estado normal. Escutas de `demanda_itens` (edição concorrente, payload pequeno) devem ter handler **próprio e rápido**, separado do de `producao_eventos` (que dispara a recarga pesada de `posicao_atual`, ~914 kB).
- Criação de tabela/coluna/view/bucket exige credencial elevada (ver seção própria). Tabelas novas seguem o padrão: `grant all to anon, authenticated` + RLS com policy permissiva `using(true) with check(true)`.

> A mesma `codigo_peca` pode existir em **vários pedidos**, cada um com `razao_social`/tipo diferente. Sempre que buscar tipo/progresso, filtrar por **pedido + código**, não só por código.

## Regras de negócio firmadas

- **Tipo V/I** (`getTipo` em status.js): `classificacao_fiscal` ("Vendas"→V, "Industrialização"→I); fallback `razao_social` (Brasmat→V/Vendas, Mota→I/Industrialização). Lookup sempre por **pedido+código**, pegando o registro mais recente com o campo preenchido (`order=id.desc`). V = azul, I = laranja.
- **Barra de progresso** (`calcProgresso`/`renderProgresso`): `sequencial_operacao / quantidade_operacoes` do último evento. A linha do tempo é a referência — replicar o mesmo cálculo nos cards (buscar histórico + `calcProgresso`). **Se a peça tem roteiro na Engenharia**, o denominador na linha do tempo passa a ser o total de itens do roteiro (`buscarTotalRoteiro`); sem roteiro, comportamento antigo.
- **Status da peça** (`status()`): expedida, em_estoque, aprovada, aguardando, em_producao, liberada — definido por evento + processo + suboperação (constantes `SUBOP_*`).
- **O que é e o que NÃO é apontamento anômalo** (confirmado pelo usuário, 22/07/2026): **repetir a mesma operação** (peça volta ao mesmo processo noutro momento — retrabalho, reentrada, ajuste) é **normal**, são apontamentos legítimos distintos. **`sequencial_operacao` maior que `quantidade_operacoes`** (ex.: "op. 18ª de 17") também é **normal** — a peça pode receber operação além do roteiro. Não tratar nenhum dos dois como erro nem "corrigir" na exibição. Anomalia real é só o **mesmo registro gravado 2x**, com todos os campos idênticos **inclusive `data_evento` até o milissegundo** (dois `id` guardando o mesmo apontamento) — critério: `pedido + codigo_peca + processo + evento + operador + data_evento`. Causa raiz corrigida no script do Forms (`LockService`); zero duplicatas novas desde 04/07/2026. Detalhes e números na skill `auditar-apontamentos`.
- **Inspeção de qualidade**: processo `QUALIDADE` tem duas suboperações — `SUBOP_INSP_PROC` ("10.1", inspeção de processo, hoje aponta entrada+saída) e `SUBOP_INSP` ("10.2", inspeção final, hoje só aponta saída — não usar para cálculo de tempo). Contagens de produtividade devem considerar só eventos de **Saída** para não contar 2x a inspeção de processo.
- **OTD** (`otd.html`): filtro **Vendas/Industrialização/Todos** (default Vendas — o prazo de venda é compromisso com o cliente; o de industrialização é previsão interna). Número-título **OTD global** (entrega final Saída ADM 11.4 vs prazo): "entregue no prazo" (÷ entregues) + "incluindo atrasos em aberto" (÷ entregues + vencidos não entregues). Abaixo, cascata por etapa (Planejamento→Produção→Qualidade→Expedição) como diagnóstico + ranking por cliente. Etapa **Estoque** removida (apontamento de entrada no estoque deixou de existir). Detalhes de cálculo na skill `kpi-definicoes`.
- **Ícones de anexo** na linha do tempo (`renderAnexosIcons`): 📐 Desenho / 📋 Roteiro só aparecem se houver PDF cadastrado na demanda daquela peça (lookup por pedido+código).
- **Kanban**: peças que estão em alguma demanda ficam com card de **fundo vermelho claro** (`.card-demanda`).
- **Demanda Cliente**: cliente vem da base; busca de item por código/pedido/OS (igual Buscar Peça) com opção "item novo" só quando não acha nada; **salvamento automático** (sem botão, debounce); check manual **"Concluído"** por item (sobrepõe o progresso quando apontamento está errado); ao remover anexo / trocar PDF / remover item, o arquivo é **apagado do bucket** (`supaDeleteArquivo`).

## Convenções de trabalho

- **Edições cirúrgicas**: mexer só no que foi pedido. Não refatorar, renomear ou "melhorar" o entorno sem pedir. O usuário preza muito o que já funciona.
- **Commit + push a cada alteração** concluída (o usuário acompanha pelo deploy). Mensagens de commit em inglês.
